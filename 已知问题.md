# 已知问题

## 可搜索PDF文字偏移

### 表现
- 生成的 `*_searchable.pdf` 中 OCR 文字层位置偏移，搜索高亮显示在空白处
- 简单文档（发票、验收单）偏移较小，复杂版面（合同、带旋转的扫描件）偏移明显
- 文字内容正确（能搜到），只是位置不对

### 当前状态
- CJK 字体嵌入 ✅
- 文字可搜索 ✅
- 文字位置 ❌ 待修复

### 已尝试的方案

#### 方案A：直接在原 PDF 上叠文字（当前部署版本）
- PaddleOCR 接收原始 PDF，内部渲染 → 返回 OCR 坐标
- Formatter 打开原 PDF，缩放 OCR 坐标后叠文字
- **问题**：PaddleOCR 内部渲染和 PyMuPDF 打开页面的坐标系不一致

#### 方案B：PyMuPDF 渲染 PNG → 发图给 PaddleOCR → 同图做底图（最新尝试）
- ocr.py：PyMuPDF 渲染 PDF 每页为 PNG（144 DPI）→ 作为图片发送给 PaddleOCR
- pdf.py：用同样的 PNG 做底图 → 1:1 叠文字
- **结果**：仍有偏移，说明 PyMuPDF `get_pixmap` 渲染的图片和 PaddleOCR 对图片的处理方式仍有差异

### 根因分析

通过 Web 搜索结果（2025-06-18）：
1. **PaddleOCR-VL 内部处理流程**：接收图片后，会经过预处理（缩放、裁剪、纠偏等），`block_bbox` 坐标是预处理后的坐标，不是原图坐标
2. **PaddleOCR 参数**：`useDocOrientationClassify`、`useDocUnwarping` 等参数会影响坐标输出
3. **DPI 问题**：PDF 渲染时点阵尺寸与逻辑尺寸的转换可能引入偏差

### 后续解决方向

| 方向 | 思路 | 复杂度 |
|------|------|--------|
| 1. 关掉 PaddleOCR 预处理 | 尝试设置 `useDocOrientationClassify=false` 等参数 | 低 |
| 2. 改用 MinerU 坐标 | MinerU 的 content_list 可能提供更精确的坐标 | 中 |
| 3. 引入 OCRmyPDF | 用专门的 OCRmyPDF 库替代自研可搜索PDF逻辑 | 中 |
| 4. 坐标校正 | 对 bbox 做实验性偏移校正（按比例平移） | 低 |
| 5. 放弃 PaddleOCR-VL | 改用 Tesseract + 版面分析或其它 OCR 引擎 | 高 |

### 复现步骤
1. 上传一份带旋转的扫描 PDF（如 270° 或 90° 旋转的合同）
2. 选择 PDF 格式输出
3. 下载生成的 `*_searchable.pdf`
4. Ctrl+F 搜索文字，观察高亮位置与原文是否对齐
