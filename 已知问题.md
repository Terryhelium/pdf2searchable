# 已知问题

## 可搜索PDF文字偏移

### 表现
- 生成的 `*_searchable.pdf` 中 OCR 文字层位置偏移，搜索高亮显示在空白处
- 旋转扫描件最明显：阅读器里看起来是竖版 A4，但底层页面和底图实际上是横向存储
- 长文本/密集文本块文档（如验收单）即使只有 1 页，也可能出现块级叠字失败
- 部分文档是“能搜到但高亮不对”，另一部分则可能是“部分文字层缺失”

### 当前状态
- CJK 字体嵌入 ✅
- 文字可搜索 ✅
- 文字位置 ❌ 待修复

### 已尝试的方案

#### 方案A：直接在原 PDF 上叠文字（当前部署版本）
- PaddleOCR 接收原始 PDF，内部渲染 → 返回 OCR 坐标
- Formatter 打开原 PDF，缩放 OCR 坐标后叠文字
- **问题**：PaddleOCR 内部渲染和 PyMuPDF 打开页面的坐标系不一致

#### 方案B：PyMuPDF 渲染 PNG → 发图给 PaddleOCR → 同图做底图（最新尝试）
- ocr.py：PyMuPDF 渲染 PDF 每页为 PNG（144 DPI）→ 作为图片发送给 PaddleOCR
- pdf.py：用同样的 PNG 做底图 → 1:1 叠文字
- **结果**：仍有偏移，说明 PyMuPDF `get_pixmap` 渲染的图片和 PaddleOCR 对图片的处理方式仍有差异

#### 方案C：代码侧保守修正（2026-06-19，待服务器验证）
- 在 `backend/app/formatters/pdf.py` 增加旋转页坐标映射逻辑
- 修正 `insert_textbox()` 失败判定，不再把严重负值当作可接受结果
- 给长文本块增加 fallback 叠字，避免静默丢字
- 在 `backend/app/tasks.py` 里补齐失败错误落库，便于前端和数据库排障
- **状态**：本地仅完成静态检查，尚未在 OCR 服务器环境联机验证

### 根因分析

通过 Web 搜索结果（2025-06-18）：
1. **旋转元数据问题**：异常样本已确认是“横向底图 + `/Rotate 90/270` 扶正显示”，肉眼看竖版不代表底层坐标系是竖版
2. **PaddleOCR-VL 内部处理流程**：接收图片后，会经过预处理（缩放、裁剪、纠偏等），`block_bbox` 坐标不一定严格对应原始 PDF 页坐标
3. **长文本块问题**：当前自研双层 PDF 方案按块把整段 `block_content` 塞进 `block_bbox`，字数多时 `insert_textbox()` 容易失败
4. **DPI/渲染问题**：PDF 渲染时点阵尺寸与逻辑尺寸的转换仍可能引入附加偏差

### 后续解决方向

| 方向 | 思路 | 复杂度 |
|------|------|--------|
| 1. 服务器验证当前保守修正 | 先确认旋转映射和长文本 fallback 是否明显改善 | 低 |
| 2. 关掉 PaddleOCR 预处理 | 尝试设置 `useDocOrientationClassify=false` 等参数 | 低 |
| 3. 改用 MinerU 坐标 | MinerU 的 content_list 可能提供更精确的坐标 | 中 |
| 4. 引入 OCRmyPDF | 用专门的 OCRmyPDF 库替代自研可搜索PDF逻辑 | 中 |
| 5. 坐标校正 | 对 bbox 做实验性偏移校正（按比例平移） | 低 |
| 6. 放弃 PaddleOCR-VL | 改用 Tesseract + 版面分析或其它 OCR 引擎 | 高 |

### 复现步骤
1. 上传一份带旋转元数据的扫描 PDF（如 270° 或 90° 旋转的合同/验收单）
2. 选择 PDF 格式输出
3. 下载生成的 `*_searchable.pdf`
4. Ctrl+F 搜索文字，观察高亮位置与原文是否对齐
